# 📚 代理方案优先序 (Roadmap)

> **核心选型建议**: 本文档旨在帮助用户根据自身的使用场景（流动场所 vs 固定场所）、设备数量、预算及技术折腾能力，选择最合适的代理方案。

---

## 1. 基础需求
> **场景**: 流动场所 ➕ 单设备

在此场景下，无需复杂的软硬路由方案。请直接参照 **[📦 按核心分类](../cores/)** 或 **[📱 移动端/💻 电脑端清单](../README.md)**，根据您手中的终端设备及操作系统选择对应的 App 即可。

---

## 2. 甜点需求 (Sweet Spot)
> **场景**: 固定场所 ➕ 长时间开启 ➕ ≥2台设备  
> **心态**: 愿花一下午时间研究，但认为 **易用性 ＞ 极致体验**

在此场景下，推荐使用旁路由或硬路由插件方案。以下是几种主流方案的对比：

### 🛜 华硕路由器 ＋ 代理插件 (★★★ 强烈推荐)
* **预算**: ￥400 ～ 2000 元
* **评价**: 对于绝大部分人而言，它接近 **“功能性 ✖️ 易用性 ✖️ 预算”** 的最佳平衡点。
* **优点**:
    * 刷官改/梅林固件、装插件属于“有手就会”。
    * WiFi 6 及之后机型 CPU 足够跑满 ⚡≥1.5Gbps 的 **单线程** 代理。
    * 扩展性强：Samba、多种内网穿透、DDNS、OpenList、Node.js、RustDesk Server 等。
    * 插件免费（虽然路由器本体比小米稍贵）。
* **缺点**:
    * 插件仅能安装到官改/梅林固件。
    * 中低端机型 ROM 仅 256MB，若不插 U 盘扩容，插件数量受限。

### 🍏 Mac Mini ＋ Surge/Stash for Mac (★★☆)
* **预算**: ￥3000 元 (起)
* **模式**: 旁路网关 (Sidecar Gateway)
* **评价**: 更昂贵的方案，提供 GUI Dashboard 对新手友好，但投入回报率极低。
* **优点**:
    * 通用 iOS 版配置文件。
    * 国补后的 Mac Mini (M4 万兆版) 本身属于 **最便宜** 的万兆软路由硬件。
* **缺点**:
    * 开启代理后的网络实际体感，相对华硕/小米硬路由基本拉不开差距。
    * 仅支持 fake-ip，搭配游戏加速器、局域网备份、投屏等 App 时易出奇奇怪怪的问题。
    * **证书风险**: 基于 MitM 去广告等功能可能导致局域网内 **非苹果设备** (不信任用户自签证书) 加载网页出错。仅适合全苹果设备家庭！

### 🛜 小米/红米路由器 ＋ 代理插件 (★★)
* **预算**: ￥120 ～ 1000 元
* **评价**: 物美价廉，丰俭由人。
* **优点**:
    * 多数 WiFi 6 及之后机型 CPU 够跑满千兆 **单线程** 代理（MT7981 机型如 AX3000T/WR30U 除外）。
* **缺点**:
    * 安装插件必须先解锁 SSH，操作比华硕、Apple TV 繁琐。
    * 中低端机型 ROM 仅 128MB，官方固件下空间仅够装 ShellCrash。
    * 不刷 OpenWrt 的话，扩展性不如华硕和 Apple TV。

### 🍎 Apple TV (有线) ＋ 一类/二类 Apps (★)
* **预算**: ￥1000 元
* **模式**: 旁路网关
* **评价**: 稳定性不如前三者，属于“博主推荐但私下不用”的非主流方案。若非刚需 Apple TV 电视功能，不值得“为醋包饺子”。
* **缺点**:
    * tvOS 对代理 App 运存限制 15MB（远低于 iOS 的 50MB），MitM 脚本非常局局促。
    * 扩展性不如华硕或刷 OpenWrt 的小米。
    * 仅支持 fake-ip，局域网备份/投屏易出问题。
    * 同样存在 MitM 证书导致非苹果设备上网出错的问题。

> **注**: 上述二星 (★★) 及以上方案足以覆盖 90% 的用户群体。

## 3. 进阶需求 (Advanced)
> **场景**: 追求更极致、更客制化的体验

如果您有以下需求，则属于进阶用户，需要更专业的 **OpenWrt / Linux 软路由** 方案：

1.  **远距离单线程 ≥1000Mbps**:
    * 仅武汉、合肥等极少数“三边形城市”可忽略。
    * **瓶颈**: 在类 Unix 默认参数下，若距离机场入口 ＞约 25ms，单线程可能卡在 ～650Mbps 瓶颈。
    * **解法**: 刚需 OpenWrt/Linux 进行 **TCP 调优 (打鸡血)**。
2.  **挑剔的“国内 ➕ 国外”双重极致体感**:
    * Surge/Mihomo/Stash 在极致延迟上往往“两难全”（即使牺牲 fake-ip）。
    * **解法**: 刚需 **Sing-box 核心** 或 爱快/ROS 的“伪多拨回环分流”。
3.  **多拨 / 宽带聚合**: 需要适配该场景的分流代理。
4.  **极客心态**: 不管能否用上，所有性能指标都要 **最极致**。

## ⚠️ 重要说明与补充

### 1️⃣ 关于 Sing-box 的极致体感
若追求最极致的体感，基于 **sing-box 核心** 的插件/客户端是不失为一种相对便捷的方案。
* **核心差异**: 迄今所有平台的各类代理软件，在 DNS 处理上仅有 "Singbox" 和 "其他" 之分。
* **体感比喻**: 其差距类似 120Hz 高刷屏 vs 60Hz 低刷屏。
* **对比**: 虽然“伪多拨回环分流”效果可能更进一步，但 Sing-box 方案实现难度远低于前者。

### 2️⃣ 决定性因素
不管选择何种软硬件方案，在今天，制约延迟、稳定性的 **最大短板** 依然在于 **节点质量**。
